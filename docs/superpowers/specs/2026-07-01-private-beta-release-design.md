# 语境生词本 Private Beta 发布设计

日期：2026-07-01  
状态：已确认设计，等待用户审阅

## 1. 目标

将现有 Chrome MV3 扩展打包为 `0.1.0` Private Beta，通过 Chrome Web
Store 的 Private 可见性邀请 5–20 名测试者进行 14 天内测。

本轮同时验证：

1. 用户是否愿意在第二周继续使用；
2. 用户是否愿意为产品付费；
3. 核心流程是否稳定。

持续使用是主判断，付费意愿和稳定性用于决定下一阶段方向。

## 2. 范围

### 2.1 包含

- Private Beta manifest、版本和商店文案；
- Chrome Web Store 上传 ZIP；
- GitHub Pages 公开隐私政策；
- 商店截图、图标和小型宣传图；
- 测试者安装与使用指南；
- 14 天微信群运营手册；
- Day 7 脉冲问卷和 Day 14 最终问卷；
- 发布检查、故障分级、版本记录和回滚说明。

### 2.2 不包含

- 账号、登录、云同步、支付或订阅；
- 插件内反馈提交；
- 分析、遥测、广告或新的外部 API；
- Public 或 Unlisted 分发；
- 与 Private Beta 无关的产品重构。

## 3. 发布架构

### 3.1 扩展运行架构

扩展继续保持本地优先：

- 词典从扩展包内的字典分片读取；
- 生词、原句、来源、设置和周报保存在本机扩展存储；
- 不向开发者或第三方服务器发送浏览内容；
- 网页访问权限继续通过 optional host permissions 由用户主动授予；
- 微信群和问卷承担反馈闭环，扩展本身不新增联网反馈能力。

### 3.2 Beta 标识和版本

- 首个商店版本为 `0.1.0`；
- manifest 名称使用“语境生词本 BETA”；
- manifest 描述明确写明该扩展用于 Beta 测试；
- 每次重新上传 Chrome Web Store 都递增 manifest 版本；
- 转为正式版本时移除 Beta 标识并再次递增版本。

### 3.3 分发

- Chrome Web Store 可见性设为 Private；
- 仅加入测试名单的 Google 账号可以查看和安装；
- 初始招募目标为 12 人，最少有效样本 8 人，最多 20 人；
- 不通过群文件分发 CRX，也不把“加载已解压扩展”作为正式内测安装方式；
- 加载已解压扩展只用于发布者自己的发布前冒烟测试。

### 3.4 隐私政策

- GitHub Pages 从仓库 `docs/` 目录发布公开 HTTPS 页面；
- 页面内容与仓库根目录 `PRIVACY.md` 保持一致；
- 页面说明收集、使用、存储、共享、导出和删除数据的方式；
- Chrome Web Store Privacy practices 中填写该公开 URL；
- 发布前从未登录浏览器访问 URL，确认页面无需权限且正文可读。

## 4. 发布产物

### 4.1 扩展 ZIP

通过 `pnpm zip` 生成 Chrome Web Store 上传包。发布检查必须确认：

- ZIP 根目录直接包含 `manifest.json`；
- manifest 为版本 3，名称、描述和版本正确；
- ZIP 只包含扩展运行文件；
- ZIP 不包含源码、测试、报告、运营文档、问卷、商店素材或秘密文件；
- 记录 ZIP 的 SHA-256、源码提交号和生成日期。

### 4.2 商店文案

仓库保存可直接粘贴到 Developer Dashboard 的文案：

- 单一用途说明；
- 简短描述和详细描述；
- `storage`、`alarms`、`notifications`、`tts`、`sidePanel`、
  `scripting` 及 optional host permissions 的逐项理由；
- 远程代码声明为“不使用”；
- 数据使用声明与隐私政策保持一致；
- Private Beta 测试说明。

### 4.3 商店素材

- 使用现有 128×128 PNG 作为商店图标，并检查明暗背景可读性；
- 至少制作 3 张 1280×800 截图：
  1. 网页悬停查词和原句高亮；
  2. 侧边栏生词列表和当前单词；
  3. 周报、设置和数据管理；
- 制作 1 张 440×280 小型宣传图；
- 截图使用演示页面和虚构内容，不显示真实浏览记录、账号或私人原句；
- 商店素材不打入扩展 ZIP。

Chrome Web Store 的 ZIP、listing 和素材规格分别以官方的
[Prepare your extension](https://developer.chrome.com/docs/webstore/prepare)、
[Complete your listing information](https://developer.chrome.com/docs/webstore/cws-dashboard-listing)
和 [Supplying Images](https://developer.chrome.com/docs/webstore/images) 为准。

## 5. 14 天内测流程

### 5.1 招募前

- 收集测试者用于 Chrome Web Store 的 Google 账号；
- 确认测试者使用 Chrome 桌面版，并记录操作系统；
- 明确本轮免费、持续 14 天、反馈通过微信群和问卷完成；
- 提醒测试者不要在群内发送包含私人网页内容的截图或导出文件。

### 5.2 Day 0：激活

群内发送安装指南，要求完成：

1. 从 Private 商店页面安装；
2. 打开侧边栏；
3. 主动授予网页权限；
4. 在普通英文文章中悬停查词；
5. 收藏至少 3 个词；
6. 确认生词可在侧边栏查看。

当天只处理安装、权限和核心流程阻塞，不引导测试者评价长期价值。

### 5.3 Day 3：阻塞检查

用一次群内短询问收集：

- 是否仍能正常查词和收藏；
- 是否遇到误触发、无法发音或侧边栏异常；
- 是否存在影响继续使用的问题。

P0、P1 问题单独登记，避免让群聊数量替代问题严重度。

### 5.4 Day 7：周报检查

发送不超过两分钟的脉冲问卷，确认：

- 过去一周使用天数；
- 是否看到并打开周报；
- 最常用和最没价值的功能；
- 是否出现数据、权限或页面兼容问题。

### 5.5 Day 14：最终评估

最终问卷收集：

- 第二周是否仍在使用；
- 14 天内总使用天数区间；
- 核心流程和周报完成情况；
- 继续使用意愿；
- 付费意愿、偏好一次性购买还是订阅、可接受价格；
- 最重要的一个价值点和一个阻塞点；
- 是否愿意参加 15 分钟访谈。

选择高频使用者、流失者和遇到严重问题者做补充访谈，避免只听满意用户。

## 6. 数据边界

- 扩展不新增遥测或反馈网络请求；
- 指标来自测试者自报、微信群问题登记和问卷；
- 不默认索要扩展导出的 JSON；
- 不收集真实原句、完整网址或浏览历史；
- 需要故障截图时，要求测试者遮盖账号、网址、输入内容和私人文本；
- 问卷只收集完成内测所需的最少信息；
- 内测结束后汇总去标识结果，原始问卷仅由发布者保管。

## 7. 成功指标

有效样本定义为完成 Day 0 核心流程并提交 Day 14 问卷的测试者。

### 7.1 激活

- 至少 80% 的受邀测试者完成 Day 0 核心流程。

### 7.2 持续使用

- 至少 60% 的有效样本在第二周仍使用；
- 上述用户在 14 天内自报使用不少于 3 个不同日期；
- 这是本轮继续推进的首要指标。

### 7.3 周报

- 至少 50% 的有效样本看到并打开过一次周报。

### 7.4 稳定性

- P0 问题为 0；
- 至少 80% 的有效样本可以完成授权、查词、收藏和查看生词；
- P1 问题不影响剩余测试者继续完成内测。

### 7.5 付费意愿

- 至少 30% 的有效样本明确表示愿意付费；
- 问卷同时收集一次性购买、订阅和可接受价格，不在本轮实现支付。

### 7.6 决策

- 持续使用达标且 P0 为 0：进入更大规模测试或付费方案设计；
- 持续使用未达标：先验证使用场景和核心价值，不增加商业化功能；
- 持续使用达标但付费意愿未达标：调整定位、价值表达或价格假设；
- 稳定性未达标：暂停扩量，先修复核心流程。

## 8. 发布门槛

每次打包上传前依次执行：

```bash
pnpm install --frozen-lockfile
pnpm typecheck
pnpm test
pnpm build
pnpm test:e2e
pnpm zip
```

自动检查通过后，在当前 Chrome/macOS 手工完成：

- 加载生产构建；
- 安装和首次打开；
- 网页权限授予、撤销和重新授予；
- 普通网页查词、收藏和发音；
- 输入框、可编辑区域和代码区域不触发；
- 侧边栏当前词、生词列表、周报和设置；
- 导出、导入和清空数据；
- 未产生外部网络请求。

`docs/manual-test-2026-06-30.md` 中的跨版本和 Windows 项在 Private Beta
期间由测试者补充。当前 Chrome/macOS 冒烟测试是首次提交的硬门槛；缺少
Windows 或 previous stable 环境不会阻止首次 Private 提交，但必须记录为
尚未覆盖的兼容性风险。

## 9. 故障处理

### 9.1 分级

- P0：隐私泄露、数据丢失、采集密码框/输入框/可编辑区域内容。立即暂停
  内测、停止新增测试者，并下架或停止分发受影响版本。
- P1：无法安装、无法授权、查词或收藏核心流程不可用。48 小时内提交最小
  修复并递增版本。
- P2：样式、文案、非阻塞兼容或低频边界问题。登记后批量处理。

### 9.2 修复原则

- 先复现并确认根因；
- 热修复只处理当前故障，不夹带新功能；
- 修复后重新执行全部发布门槛；
- 在群内说明影响范围、临时规避方式和新版本状态；
- 不要求测试者公开发送私人数据来证明故障。

## 10. 版本记录与回滚

每次商店提交保存：

- manifest 版本；
- Git 提交号；
- ZIP 文件名和 SHA-256；
- 自动和手工检查结果；
- 提交日期、审核状态和发布时间；
- 已知风险。

发生严重问题时：

1. 立即停止新增测试者；
2. 若商店支持且存在已验证旧版本，回滚到旧版本；
3. 首版无旧版本可回滚时，暂时取消发布或下架；
4. 提交只包含必要修复的更高版本；
5. 复验通过后恢复邀请。

## 11. 计划中的文件边界

实施阶段优先使用以下边界，避免发布资料散落：

- `wxt.config.ts`：Beta manifest 元数据；
- `docs/privacy/`：GitHub Pages 隐私页面；
- `docs/release/`：商店文案、权限说明、测试者指南、运营手册、问卷和发布记录模板；
- `store-assets/`：不进入扩展 ZIP 的商店截图和宣传图；
- `docs/superpowers/plans/`：后续实施计划。

## 12. 验收标准

设计实施完成的判定条件：

- 所有发布门槛通过；
- ZIP 内容和 SHA-256 已记录；
- GitHub Pages 隐私政策可公开访问；
- Developer Dashboard 所需文案和权限说明可直接使用；
- 图标、至少 3 张截图和 440×280 宣传图就绪；
- 测试者指南、Day 7/Day 14 问卷和 14 天运营手册就绪；
- Private 测试名单和首批 12 名招募目标明确；
- 未引入遥测、反馈 API、支付或云端用户数据存储。
